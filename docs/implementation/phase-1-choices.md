# Phase 1 Implementation Choices

**Date:** 2026-07-12
**Branch:** `feature/stage-1-schema-store`
**Status:** accepted for Phase 1 implementation

This document records the six deliberate implementation choices made before
coding the schema + store foundation. Each choice was identified during the
repo orientation sanity check and resolved here before writing code.

## 1. ID Scheme

**Decision:** `rec_<type>_<ulid>`

- ULID is 26 characters, sortable (time-ordered), URL-safe, and case-insensitive.
- The `rec_` prefix and `<type>` segment make IDs human-greppable in CLI output.
- Example: `rec_task_01HZX4F8GHNQKZPQ7TQ3R8V1ZM`.
- IDs are generated by Zentext on create, immutable thereafter.
- Uses the `ulid` npm package for generation.

**Rationale:** The docs said "final scheme decided in implementation; must be
stable, unique, and human-greppable." ULID gives sortable + unique; the
`rec_<type>_` prefix gives human-readability.

## 2. SQLite Migration Approach

**Decision:** `PRAGMA user_version` with ordered in-code migration functions.

- Each migration is a function that takes a `better-sqlite3` Database instance.
- Migrations run in order: 1, 2, 3, ...
- `PRAGMA user_version` tracks which migrations have been applied.
- On open: read `user_version`, run all migrations with a higher number.
- Initial schema is migration #1 (creates `records`, `record_history`, `meta`
  tables).
- Future migrations increment `user_version` and are appended to the list.

**Rationale:** Minimal, no external dependency, fits single-file local SQLite.
No need for a migration tool at Stage 1.

## 3. Store API Abstraction Boundary

**Decision:** Define a `Store` interface in `src/types/store.ts`. The concrete
`SqliteStore` class implements it in `src/store/sqlite-store.ts`. All other code
depends on the `Store` interface, not `SqliteStore` directly.

Minimum interface (as specified by the user):
- `initProjectStore` — create the store directory and database if absent.
- `openProjectStore` — open an existing store.
- `createRecord` — create a new record, generate fields, validate.
- `getRecord` — read a single record by id.
- `listRecords` — list records by project, optionally filtered by type/status.
- `updateRecord` — update a record, increment revision, write history event.
- `close` — close the database connection.

The interface does not over-abstract for cloud, sync, or distributed
coordination. It is abstracted enough that the backend could be swapped if
Stage 2 needs it, per ADR 0002 risk mitigation.

## 4. SQLite Record Storage Format

**Decision:** Hybrid — relational columns for queryable common fields + JSON
column for type-specific payload + separate `record_history` table.

`records` table columns:
- `id` TEXT PRIMARY KEY
- `project_id` TEXT NOT NULL
- `type` TEXT NOT NULL
- `title` TEXT NOT NULL
- `status` TEXT NOT NULL
- `summary` TEXT
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL
- `revision` INTEGER NOT NULL DEFAULT 1
- `author` TEXT NOT NULL
- `tags_json` TEXT (JSON array of strings)
- `refs_json` TEXT (JSON object)
- `supersedes_json` TEXT (JSON array, nullable)
- `superseded_by` TEXT (nullable)
- `schema_version` INTEGER NOT NULL DEFAULT 1
- `payload_json` TEXT (JSON — type-specific content)

`record_history` table columns:
- `id` INTEGER PRIMARY KEY AUTOINCREMENT
- `record_id` TEXT NOT NULL
- `revision` INTEGER NOT NULL
- `event` TEXT NOT NULL (`create`, `update`, `supersede`)
- `occurred_at` TEXT NOT NULL
- `author` TEXT
- `record_json` TEXT NOT NULL (full record snapshot for that revision)

`meta` table:
- `key` TEXT PRIMARY KEY
- `value` TEXT

**Rationale:** Common fields in relational columns enable efficient SQL queries
(by type, status, project, etc.). Type-specific content in `payload_json` keeps
the schema flexible without a wide table or dozens of nullable columns. History
in a separate table keeps the current-state table lean.

## 5. Status Omitted/Null/Default Behavior

**Decision:**
- **Omitted** status on create: use the type-specific default (from the schema
  contract).
- **Explicit `null`** status on create: rejected with a validation error.
- **Unknown status value** (not in the allowed enum for that type): rejected
  with a validation error.
- **Omitted on update**: status is unchanged (not reset to default).

Default statuses per type (from `data-model-and-store.md`):

| Type | Default status |
|------|----------------|
| `task` | `active` |
| `decision` | `accepted` |
| `blocker` | `open` |
| `handoff` | `latest` |
| `log` | `recorded` |
| `validation` | same as `result` field |
| `policy` | `active` |
| `custom` | `active` |

For `validation`: if `status` is omitted, it defaults to the value of `result`
(which is required). If both `status` and `result` are provided and disagree,
`result` wins for the default; explicit `status` is honored.

**Rationale:** Makes behavior explicit and predictable. Prevents accidental
null records while being developer-friendly with sensible defaults.

## 6. Author Field Resolution

**Decision:**
- The store API accepts an optional `author` string from the caller.
- If omitted, the store defaults to `"unknown"`.
- MCP/CLI layers (future phases) will pass better author values
  (`agent:codex`, `user:judah`, `ci:github-actions`, etc.).
- Phase 1 does not try to detect agent identity automatically.

**Rationale:** The store layer should not guess who is writing. The caller
knows its identity. The store provides a safe default so records are never
missing an author field. This matches the docs' `author` enum format while
keeping Phase 1 simple.
