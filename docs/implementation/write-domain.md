# Zentext Write Domain

**Status:** Stage 2A implementation.

The write domain is the single canonical layer for all Zentext memory
mutations. CLI write commands and MCP write tools will be thin adapters over
this layer. Nothing outside the domain layer should write directly to the store
or manually create revision history.

## Core principle

One canonical implementation, multiple delivery surfaces.

## Mutation entry points

- `createRecord` — create a new canonical record.
- `updateRecord` — patch an existing record with optimistic concurrency.
- `supersedeRecord` — replace an existing record with a new one.
- `archiveRecord` — mark a record as archived/inactive.
- `createHandoff` — add a handoff record and mark prior handoffs as archived.

## Invariants

### Identity

- Record IDs are generated by the domain layer; callers cannot choose IDs.
- Every record belongs to exactly one project.
- Cross-project mutation is impossible because the domain service is bound to
  an already-open project store.

### Revision history

- Creation produces revision 1.
- Every successful mutation increments revision exactly once.
- Revision numbers are contiguous.
- History is stored oldest-to-newest by `revision` ascending.
- Each history entry records the full resulting record state and the event type
  (`create`, `update`, `supersede`, `archive`, `handoff`).
- Failed or no-op mutations do not increment revisions.
- Current canonical revision matches the latest history entry.

### Status

- Only statuses valid for the record type may be stored.
- Invalid status transitions are rejected at the domain boundary.
- Archived and superseded records are not treated as current truth by the
  repack engine.

### Supersession

- A replacement record must be created for the same project.
- The source record must exist and must not already be superseded.
- The replacement record links back to the source via `supersedes`.
- The source record is atomically marked `superseded_by`.
- A record cannot supersede itself.
- Cycles are impossible because a new record cannot supersede a record that is
  already superseded.

### Updates

- Record type and project identity cannot change.
- Immutable fields remain immutable.
- Omitted fields remain unchanged.
- Passing `summary: null` clears the summary.
- Updating a missing, archived, or superseded record is rejected.
- No-op updates (no fields changed and expected revision matches current) return
  the unchanged record without creating a new revision.

### Archive

- Archive is idempotent: archiving an already-archived record returns the record
  without a new revision.
- Archiving preserves history.
- Archiving does not delete the record.
- Superseded records cannot be archived.

### Handoff

- A handoff requires `from`, `to`, `context`, `state`, and `next`.
- Creating a handoff automatically marks any existing `latest` handoff as
  `archived` so that `latest` is always unambiguous.

## Optimistic concurrency

`updateRecord`, `archiveRecord`, and `supersedeRecord` accept an optional
`expectedRevision`. If supplied and it does not match the current revision, the
operation is rejected with a `MemoryWriterConflictError` and makes no persistent
change.

## Domain errors

All errors extend `MemoryWriterError`. Specific classes include:

- `MemoryWriterNotFoundError` — record or referenced record missing.
- `MemoryWriterConflictError` — optimistic revision conflict.
- `MemoryWriterValidationError` — invalid status, field, or transition.
- `MemoryWriterStateError` — operation not allowed in current record state
  (e.g., already superseded, archived).

## Transaction strategy

The domain service relies on SQLite transactions inside the store. Each mutation
is wrapped in a transaction that either commits both the canonical record change
and its history entry, or rolls back entirely. If the store abstraction does not
expose a transaction boundary, the domain layer uses the store's internal
`withTransaction` helper.

## Store boundary

Adapters must call the write domain. They must not call `store.createRecord`,
`store.updateRecord`, or `store.getRecordHistory` directly. Low-level store
methods exist only to serve the domain layer and read-only adapters.
